问题一： LiveData 为粘性事件

首先，从安大通App开发中实际上遇到的问题引入，我们在使用LiveData处理事件的时候出现了一些问题，那就是明明我对某一事件进行 observe 后，会无可避免的收到之前设置的值。这就是因为LiveData所发出的事件是粘性事件。

当我们调用 observe方法 我们的Observer实现类将被包装成LifecycleBoundObserver后添加到 mObservers，并且添加wrapper为 Lifecycle的观察者

可见 LifecycleBoundObserver类实现了 LifecycleEventObserver类的 onStateChanged方法，所以当 LifecycleOwner 的声明周期发生变化 onStateChanged方法就会被调用， 其中的 activeStateChanged(shouldBeActive());就是关键了，这里传入的布尔值是判断生命周期状态：当前状态在STARTD之前为false，在STARTD状态之后为true。

可见 当传入值为 true 时 调用了 dispatchingValue方法，传入了 LifecycleBoundObserver类对象本身

在 considerNotify函数中已经很明显了，当 observer.mLastVersion < mVersion（LiveData的数据版本）就会调用我们的Observer的 onChanged 方法。
那么问题来了，mLastVersion 和 mVersion** ** 初始值是多少？mVersion又是如何改变的？
首先 mLastVersion 和 mVersion** ** 初始值是多少？

可见，二者开始的值都是 -1 但是如果通过LiveData的有参构造创建那么mVersion为 0，任何Observer观察后会直接触发
第二个，mVersion又是如何改变的？

是的，通过set/postValue 方法， 每更新一次value就会让 mVersion 更新。
​

OK，源码分析到这里，我相信应该可以知道为什么会产生开头所表述的那样的问题了，也就是说在LiveData被observe之前如果被修改过值那么新observe的Observer也会收到通知。

问题二： 数据倒灌

首先 ** 数据倒灌 **一词出自KunMinX的专栏重学安卓

即在setValue后,observe对此次set的value值会进行多次消费。比如进行第二次observe的时候获取到的数据是第一次的旧数据。

看了针对数据倒灌的描述，我们基本上也就可以确定数据倒灌产生的原因了.
即数据倒灌产生的本质：LiveData 为粘性事件，如果LiveData不是粘性的也就不存在数据倒灌问题，因为observe之前的事件不会被被通知。
​

解决方案

  1. 使用重写LiveData，在observe方法中反射修改Observer的初始 Version 等于 mVersion

  2. 使用SingleLiveEvent可以解决数据倒灌，但是无法解决粘性事件，而且SingleLiveEvent 的问题在于它仅限于一个观察者。如果您无意中添加了多个，则只会调用一个，并且不能保证哪一个。

  3. 对事件进行包装，添加事件被处理过的标识，可以解决数据倒灌，但是无法解决粘性事件

  4. UnPeekLiveData 不用反射技术实现LiveData的非粘性态

    个人观点

    LiveData 的粘性问题并不是Bug 而是使用场景的问题

  • LiveData 应该作为状态的储存容器，以达到界面上的UI时刻与状态保持一致
  • LiveData 不应作为事件的储存容器，改用 UnPeekLiveData 和 SingleLiveEvent 储存事件状态更为合适

Kotlin Flow 有没有这个问题？

答案是 StateFlow同样有着和 LiveData 一样的问题， 而 SharedFlow就没有
简而言之，就是下面这样：

StateFlow 类似 LiveData 作为状态的储存容器

SharedFlow 类似 UnPeekLiveData 作为事件的储存容器

Google 是如何做的？

  1. 凡是发送给 View 的事件都应该涉及 UI 变动，与 UI 无关的事件不应该由 View 监听
  2. 既然是涉及 UI 的事件，可以跟随 UI 状态一起发送（基于 StateFlow 或 LiveData ），不必另建新的渠道。
  3. View 在处理完事件后，需要告知 ViewModel 事件已处理，ViewModel 更新状态避免再次消费
